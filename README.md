# SplineCC: Алгоритм управления перегрузкой на основе кубических сплайнов

## Обзор
SplineCC — это специализированный алгоритм управления перегрузкой, реализованный на языке C (`SpCC.c`). Он предназначен для оптимизации производительности сети путем динамической настройки окна перегрузки (`cwnd`) на основе времени кругового пути (RTT), пропускной способности и количества подтверждений (ACK). В отличие от традиционных алгоритмов, таких как TCP Reno или CUBIC, SplineCC использует кубические сплайны для более плавной адаптации окна, что делает его подходящим для сетей с переменной задержкой и пропускной способностью.

Этот алгоритм идеально подходит для разработчиков и исследователей, работающих над сетевыми протоколами, особенно в условиях высокой задержки или специфических требований к производительности.

## Особенности
- **Адаптация с кубическими сплайнами**: Использует коэффициенты (`c`, `d`, `b`), вычисленные на основе RTT, `cwnd` и пропускной способности, для плавной настройки окна.
- **Фаза медленного старта**: Реализует агрессивное увеличение окна на начальном этапе, переходя к избеганию перегрузки при достижении порога (`ssthresh`).
- **Обнаружение перегрузки**: Реагирует на перегрузку сети (например, рост RTT или снижение числа ACK) путем консервативного уменьшения окна.
- **Оптимизированные вычисления**: Использует быстрые арифметические макросы (например, `MULu64_FAST`, `DIV3`) и кэширование для минимизации вычислительных затрат.
- **Обработка дублированных ACK**: Уменьшает окно и порог при обнаружении признаков потери пакетов.

## Вычисления
Функция `SplineCC` вычисляет следующее окно перегрузки (`next_cwnd`) на основе входных параметров: `curr_cwnd`, `curr_rtt`, `throughput`, `num_acks` и структуры состояния (`sCC`). Ниже приведен пошаговый разбор вычислений с примером, где:
- `curr_cwnd = 20` (сегментов), `curr_rtt = 150` (мкс), `throughput = 2000` (байт/с), `num_acks = 3`.
- Начальное состояние: `last_rtt = 100` (мкс), `last_cwnd = 15`, `ssthresh = 30`, `last_max_cwnd = 25`, `last_min_rtt = 100`, `last_ack = 2`.

### Пошаговые вычисления
1. **Инициализация**:
   - Устанавливаем `state->curr_cwnd = 20` и `state->curr_rtt = 150`.

2. **Коэффициент `d` (на основе RTT, `find_cof_rtt`)**:
   - Вычисляем соотношение текущего и последнего RTT: `(curr_rtt * 8) / last_rtt = (150 * 8) / 100 = 12`.
   - Рассчитываем кубическое соотношение: `(12 * 12 * 12) / 2 = 864`.
   - Так как `curr_rtt > last_rtt + ERR_R` (150 > 100 + 5), вычисляем `loc_rtt = 864 + (curr_rtt / 2) = 864 + 75 = 939`.
   - Вычисляем `result = loc_rtt + (loc_rtt / 2) = 939 + 469 = 1408`.
   - Устанавливаем `d = (ratio * 2) + ((result + loc_rtt) / loc_rtt) = (12 * 2) + (1408 + 939) / 939 ≈ 24 + 2 = 26`.
   - Обновляем `state->d = 26`, `state->d_initial = 26`, `state->last_rtt = 150`.

3. **Медленный старт (`handle_slow_start`)**:
   - Проверяем, выполняется ли `curr_cwnd < ssthresh`: `20 < 30`, значит, активна фаза медленного старта.
   - Обновляем ACK: `state->last_ack = 2`, `state->curr_ack = 3`.
   - Обнаруживаем перегрузку: `curr_rtt = 150 > last_min_rtt * 39 / 32 ≈ 121.875`, поэтому используем консервативный рост.
   - Увеличиваем окно: `curr_cwnd += (last_cwnd / 8) + ((num_acks + 1) / 2) = 20 + (15 / 8) + ((3 + 1) / 2) = 20 + 1 + 2 = 23`.
   - Обновляем `state->next_cwnd = 23`, `state->last_cwnd = 23`.
   - Так как `slow_start_cwnd = 23`, возвращаем это значение и пропускаем дальнейшие шаги.

4. **Корректировка порога (`ssthresh_comp`)**:
   - Вычисляем `max_ssthresh = DIV100(last_max_cwnd * THRESHOLD_PERCENT) = (25 * 100 * 3) / 256 ≈ 29`.
   - Так как `curr_ack = 3 > last_ack = 2` и `ssthresh = 30 > max_ssthresh = 29`, устанавливаем `ssthresh = 29`.

5. **Другие коэффициенты (если медленный старт не активен)**:
   - **Коэффициент `c` (`find_cof_cwnd`)**: Если `curr_cwnd >= ssthresh`, сравниваем `last_cwnd` и `curr_cwnd`. Например, при `curr_cwnd = 30`, `c = (last_cwnd + curr_cwnd) - d_initial = (18 + 30) - 26 = 22`.
   - **Коэффициент `b` (`find_cof_bw`)**: Вычисляем `throughput / (c + d_initial) = 2000 / (22 + 26) ≈ 41`, затем `b = throughput_t / throughput ≈ 48`.

6. **Следующее окно (`resolve_next_cwnd`)**:
   - Если медленный старт не активен, корректируем `next_cwnd` в зависимости от состояния сети:
     - Стабильная сеть: Увеличиваем `cwnd` на `d` и рост числа ACK.
     - Перегрузка: Уменьшаем `cwnd` на 10% (с помощью `MULT0_9`).
     - Ограничиваем `next_cwnd` двойным значением `last_max_cwnd`.

### Пример результата
Для указанных входных данных `SplineCC` возвращает `next_cwnd = 23` (фаза медленного старта) и обновляет `ssthresh = 29`.

### Ключевые аспекты
- **Кубические сплайны**: Нелинейные вычисления через `ratio_cubed` обеспечивают плавную адаптацию.
- **Быстрая арифметика**: Макросы, такие как `MULu64_FAST` и `DIVu64`, оптимизируют производительность.
- **Кэширование**: Хранит `cached_ratio` и `cached_throughput` для уменьшения повторных вычислений.
- **Ограничения**: Значения `cwnd`, RTT и `ssthresh` ограничены для предотвращения переполнения или нестабильности.

## Установка
Для использования SplineCC требуется компилятор C (например, `gcc`) и базовая среда разработки.

1. Клонируйте репозиторий:
   ```bash
   git clone https://github.com/your-username/splinecc.git
   cd splinecc
   ```
2. Скомпилируйте код:
   ```bash
   gcc -o splinecc SpCC.c
   ```
   Примечание: Возможно, потребуется интегрировать `SpCC.c` в более крупное сетевое приложение или симулятор (например, `ns-3` или пользовательский стек TCP).

## Использование
Чтобы интегрировать SplineCC в ваш проект:
1. Включите `SpCC.c` в вашу кодовую базу.
2. Инициализируйте структуру `sCC` для хранения состояния:
   ```c
   sCC state = {0}; // Инициализация всех полей нулями
   ```
3. Вызывайте функцию `SplineCC` в цикле обработки пакетов:
   ```c
   u32 next_cwnd = SplineCC(curr_cwnd, curr_rtt, throughput, num_acks, &state);
   ```
4. Используйте возвращённое значение `next_cwnd` для установки окна перегрузки в вашем протоколе.

### Пример
```c
#include "SpCC.c"
#include <stdio.h>

int main() {
    sCC state = {0};
    u32 curr_cwnd = 10;    // Начальное окно (сегменты)
    u32 curr_rtt = 100;    // RTT в микросекундах
    u64 throughput = 1000; // Пропускная способность в байтах/с
    u32 num_acks = 2;      // Количество ACK

    u32 next_cwnd = SplineCC(curr_cwnd, curr_rtt, throughput, num_acks, &state);
    printf("Следующее окно перегрузки: %u сегментов\n", next_cwnd);

    return 0;
}
```

## Конфигурация
Алгоритм использует несколько фиксированных констант, которые можно настроить для конкретных сетей:
- `MAX_RTT`: Максимальный RTT (1,000,000 мкс = 1 секунда).
- `MIN_RTT`: Минимальный RTT (10 мкс).
- `ERR_R`: Допустимая погрешность RTT (5 мкс).
- `MAX_SSHTHRESH`: Максимальный порог медленного старта (1000 сегментов).
- `THRESHOLD_PERCENT`: Процент для вычисления `ssthresh` (100%).

Для изменения этих значений отредактируйте директивы `#define` в начале файла `SpCC.c`. В будущих версиях может быть добавлена динамическая настройка через структуру `sCC`.

## Тестирование
Для проверки SplineCC:
1. **Симуляция**: Используйте сетевые симуляторы, такие как `ns-3` или `Mininet`, для тестирования производительности в различных условиях (например, высокая задержка, потери пакетов).
2. **Реальные тесты**: Интегрируйте с TCP-стеком или пользовательским протоколом и отслеживайте пропускную способность, задержки и потери пакетов.
3. **Краевые случаи**: Проверяйте поведение при экстремальных значениях (например, `curr_rtt = MIN_RTT` или `MAX_RTT`, нулевое количество ACK).

## Ограничения
- **Сложность**: Использование кубических сплайнов и множества коэффициентов делает алгоритм сложнее, чем TCP Reno или CUBIC.
- **Обработка потерь**: Ограниченная поддержка восстановления после потерь пакетов (например, Selective ACK или тайм-ауты).
- **32-битные ограничения**: Использование 32-битных целых чисел для `cwnd` и RTT может ограничивать масштабируемость в высокоскоростных сетях.
- **Фиксированные параметры**: Жёстко заданные константы могут быть неоптимальны для всех типов сетей (например, LAN с низкой задержкой или WAN с высокой задержкой).

## Планы по улучшению
- Добавить динамическую настройку параметров в зависимости от условий сети.

- Перейти на 64-битные целые числа для `cwnd` и RTT для поддержки высокоскоростных сетей.
- Упростить вычисления коэффициентов для повышения удобства поддержки.
- Интегрировать pacing для равномерного распределения отправки пакетов во времени.
