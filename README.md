# SplineCC: Алгоритм управления перегрузкой с адаптивными режимами

## Обзор
SplineCC — это современный алгоритм управления перегрузкой, реализованный на языке C (`SpCC.c`). Он разработан для оптимизации производительности сети путем динамической настройки окна перегрузки (`cwnd`) на основе времени кругового пути (RTT), пропускной способности (`throughput`) и количества подтверждений (ACK). SplineCC сочетает элементы bandwidth-delay-based (как BBR) и loss-based (как CUBIC) подходов, используя четыре режима работы для адаптации к различным сетевым условиям: `START_PROBE`, `PROBE_BW`, `PROBE_RTT` и `DRAIN_PROBE`.

Алгоритм подходит для сетей с переменной задержкой и пропускной способностью, таких как беспроводные сети или высокоскоростные WAN, и предназначен для разработчиков и исследователей, работающих над сетевыми протоколами.

## Особенности
- **Многорежимная адаптация**:
  - `START_PROBE`: Агрессивно увеличивает `cwnd` для быстрого захвата доступной полосы.
  - `PROBE_BW`: Настраивает `cwnd` на основе оценённой пропускной способности (`bw`), сглаженной через экспоненциальное скользящее среднее (EMA).
  - `PROBE_RTT`: Минимизирует задержки, корректируя `cwnd` для измерения минимального RTT.
  - `DRAIN_PROBE`: Сбрасывает избыточное окно при признаках перегрузки (высокий RTT или низкие ACKs).
- **Оценка сети**: Использует метрики `eps` (на основе RTT) и `gamma` (на основе ACKs) для оценки состояния сети и управления `cwnd`.
- **Фаза медленного старта**: Переходит в режим избегания перегрузки при достижении порога `ssthresh`.
- **Обнаружение перегрузки**: Реагирует на рост RTT или снижение ACKs, уменьшая `cwnd` на 10–30%.
- **Оптимизированные вычисления**: Использует быстрые арифметические макросы (`DIV3`, `DIV100`, `MULT0_9`) для минимизации вычислительных затрат.

## Вычисления
Функция `SplineCC` вычисляет следующее окно перегрузки (`next_cwnd`) на основе входных параметров: `curr_rtt`, `throughput`, `num_acks` и структуры состояния (`sCC`). Ниже приведён пошаговый разбор вычислений с примером, где:
- `curr_rtt = 12` мкс, `throughput = 125000000` байт/с, `num_acks = 10`.
- Начальное состояние: `last_rtt = 50` мкс, `last_cwnd = 10`, `last_max_cwnd = 120`, `last_min_rtt = 100`, `last_ack = 10`, `curr_cwnd = 64`, `ssthresh = 120`.

### Пошаговые вычисления
1. **Инициализация**:
   - Устанавливаем `state->curr_cwnd = 64`, `state->curr_rtt = 12`, `state->throughput = 125000000`.

2. **Вычисление `eps` (`__epsilone_rtt`)**:
   - Обновляем ACKs: `state->last_ack = 10`, `state->curr_ack = 10`.
   - Вычисляем `eps = ((rtt + curr_rtt) / rtt) + 1 = ((12 + 12) / 12) + 1 = 3`.
   - Обновляем RTT: `state->last_rtt = 12`, `state->curr_rtt = 12`.
   - Проверяем минимальный RTT: `last_min_rtt = 100 > curr_rtt = 12`, обновляем `last_min_rtt = 12`, увеличиваем `epp_min_rtt`.

3. **Оценка пропускной способности (`__bw`)**:
   - Вычисляем `rtt_scaling = (last_min_rtt + last_rtt) / 2 = (12 + 12) / 2 = 12`.
   - Рассчитываем `bw = (throughput * rtt_scaling) >> FIXED_SHIFT = (125000000 * 12) >> 10 ≈ 1464843`.
   - Переводим в сегменты: `state->bw = 1464843 / 1460 ≈ 1003`.
   - Применяем EMA: `state->last_bw = (last_bw * 3 + bw) >> 2 = (0 * 3 + 1003) / 4 ≈ 250`.
   - Фильтруем `bw`: `min_allowed = (last_bw * 3) >> 2 = 187`, `max_allowed = (last_bw * 6) >> 2 = 375`. Так как `bw = 1003 > max_allowed`, устанавливаем `state->bw = 375`.

4. **Вычисление `gamma` (`__gamma_ack`)**:
   - `gamma = ((curr_ack + last_ack) / curr_ack) + 1 = ((10 + 10) / 10) + 1 = 3`.

7. **Режимы работы (`probs`)**:
   - Если медленный старт не активен, вызывается `probs` с учётом режима (`START_PROBE`, `PROBE_BW`, `PROBE_RTT`, `DRAIN_PROBE`).
   - В `START_PROBE`: `curr_cwnd += curr_ack = 64 + 10 = 74`.
   - В `PROBE_BW`: `curr_cwnd` корректируется на основе `eps` и `bw`, например, `curr_cwnd = bw = 375` (если `bw <= curr_cwnd`).
   - В `PROBE_RTT`: `curr_cwnd = bw + (curr_ack >> 2) = 375 + (10 / 4) ≈ 377`.
   - В `DRAIN_PROBE`: `curr_cwnd -= (curr_cwnd - bw) / 2`.

### Пример результата
Для указанных входных данных `SplineCC` возвращает `next_cwnd = 69` (фаза медленного старта) и обновляет `ssthresh = 120`.

### Ключевые аспекты
- **Режимы работы**: Переключение между режимами каждые 10 эпох или при перегрузке (`eps <= 1`, `gamma <= 1`, высокий RTT).
- **Сглаживание `bw`**: EMA (75%) и фильтры (75–150% от `last_bw`) обеспечивают стабильность.
- **Чувствительность к RTT и ACKs**: Реакция на рост RTT (уменьшение `cwnd` на 10–30%) и снижение ACKs (через `fairness_rtt` и `fairness_rtt_bw`).
- **Быстрая арифметика**: Макросы (`DIV3`, `DIV100`, `MULT0_9`) оптимизируют вычисления.

## Установка
Для использования SplineCC требуется компилятор C (например, `gcc`) и базовая среда разработки.

1. Клонируйте репозиторий:
   ```bash
   git clone https://github.com/your-username/splinecc.git
   cd splinecc
   ```
2. Скомпилируйте код:
   ```bash
   gcc -o splinecc SpCC.c
   ```
   Примечание: Для реального использования интегрируйте `SpCC.c` в сетевой стек (например, TCP или симулятор `ns-3`).

## Использование
Чтобы интегрировать SplineCC в ваш проект:
1. Включите `SpCC.c` в вашу кодовую базу.
2. Инициализируйте структуру `sCC`:
   ```c
   sCC state = {0}; // Инициализация всех полей нулями
   state.last_rtt = 50;
   state.last_cwnd = 10;
   state.last_max_cwnd = 120;
   state.last_min_rtt = 100;
   state.last_ack = 10;
   state.curr_cwnd = 64;
   ```
3. Вызывайте `SplineCC` в цикле обработки пакетов:
   ```c
   u32 next_cwnd = SplineCC(curr_rtt, throughput, num_acks, &state);
   ```
4. Используйте `next_cwnd` для установки окна перегрузки.

### Пример
```c
#include "SpCC.c"
#include <stdio.h>

int main() {
    sCC state = {0};
    state.last_rtt = 50;
    state.last_cwnd = 10;
    state.last_max_cwnd = 120;
    state.last_min_rtt = 100;
    state.last_ack = 10;
    state.curr_cwnd = 64;
    u32 curr_rtt = 12;
    u64 throughput = 125000000;
    u32 num_acks = 10;

    u32 next_cwnd = SplineCC(curr_rtt, throughput, num_acks, &state);
    printf("Следующее окно перегрузки: %u сегментов\n", next_cwnd);

    return 0;
}
```

## Конфигурация
Алгоритм использует фиксированные константы, которые можно настроить:
- `MAX_RTT`: 1,000,000 мкс (1 секунда).
- `MIN_RTT`: 1 мкс.
- `MAX_SSHTHRESH`: 900,000 сегментов (примерно 10 Гбит/с при MSS = 1460).
- `THRESHOLD_PERCENT`: 100% для вычисления `ssthresh`.
- `FIXED_SHIFT`: 10 для фиксированной арифметики.

Для изменения отредактируйте директивы `#define` в `SpCC.c`.

## Тестирование
Для проверки SplineCC:
1. **Симуляция**: Используйте `ns-3`, `Mininet` или аналогичные симуляторы для тестирования в условиях с потерями, высокой задержкой или переменной пропускной способностью.
2. **Реальные тесты**: Интегрируйте в TCP-стек и отслеживайте метрики (пропускная способность, RTT, потери).
3. **Краевые случаи**: Проверяйте поведение при `curr_rtt = MIN_RTT`, `MAX_RTT`, или нулевых ACKs.

# SplineCC: Сравнение с другими алгоритмами

SpineCC — это алгоритм управления перегрузкой, сочетающий черты BBR и CUBIC, но обладающий уникальными особенностями. Мы протестировали реализацию SpineCC (SpineCCNew) в сравнении с BBR в двух сценариях:

1. **Конкуренция канала**: оба алгоритма используют один канал.
2. **Разные каналы**: каждый алгоритм работает в своём канале.

## Параметры тестирования
- **RTT**: 50 мс
- **Скорость канала**: 10,000 Мбит/с
- **Скорость приложения**: 5,000 Мбит/с
- **Потери**: 0.00001 (0.001%)
- **Буфер**: 1,000 пакетов (DropTail + PIE)
- **Инициализация TCP**:
  - Начальное окно перегрузки (cwnd): 10
  - MSS (Maximum Segment Size): 1,448 байт
- **Симулятор**: ns-3

## Результаты сравнения

### Пропускная способность (Throughput)
- **Конкуренция канала**:
  - **BBR**: 1.2 Мбит/с, потери — 1.22%.
  - **SpineCCNew**: 1.0 Мбит/с, потери — 1.19%.
  - **Вывод**: BBR обеспечивает более высокую пропускную способность, но SpineCCNew демонстрирует чуть меньшие потери.
- **Разные каналы**:
  - **BBR**: 1.2 Мбит/с, потери — 1.02%.
  - **SpineCCNew**: 1.1 Мбит/с, потери — 1.23%.
  - **Вывод**: BBR сохраняет лидерство по пропускной способности, но SpineCCNew улучшает свои показатели, хотя с ростом потерь.

### Окно перегрузки (CWND)
- **Конкуренция канала**:
  - **BBR**: пики до 50,000 байт, но с большими колебаниями.
  - **SpineCCNew**: более стабильное поведение, диапазон 30,000–40,000 байт.
- **Разные каналы**:
  - **BBR**: пики снижаются до 40,000 байт.
  - **SpineCCNew**: стабилизируется в диапазоне 15,000–25,000 байт, с меньшими колебаниями.
- **Вывод**: SpineCCNew демонстрирует более стабильное поведение окна перегрузки по сравнению с BBR.

### Задержки (Delay)
- **Оба сценария**:
  - **BBR**: 0.0498–0.0500 с.
  - **SpineCCNew**: 0.0500–0.0502 с.
  - **Вывод**: Задержки практически идентичны, с минимальным преимуществом BBR.

### Время кругового обхода (RTT)
- **Конкуренция канала**:
  - **BBR**: 100–140 мс.
  - **SpineCCNew**: 105–130 мс.
- **Разные каналы**:
  - **BBR**: 100–140 мс.
  - **SpineCCNew**: 105–130 мс.
  - **Вывод**: SpineCCNew обеспечивает меньшие вариации RTT, демонстрируя большую стабильность.

## Выводы
- **BBR** превосходит SpineCCNew по пропускной способности в обоих сценариях, но имеет более выраженные колебания в окне перегрузки и RTT.
- **SpineCCNew** демонстрирует стабильность в управлении окном перегрузки и RTT, а также меньшие потери в сценарии конкуренции канала.
- SpineCCNew может быть предпочтительным в сценариях, где важна стабильность и предсказуемость поведения сети.
